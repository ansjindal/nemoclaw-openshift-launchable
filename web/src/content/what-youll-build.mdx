## The idea: specialists, not clones

One sealed agent is useful. A *system* is several, each a **specialist** whose role is the skill it carries and the one host it may reach:

- a **reader** with the `@workshop/repo-reader` skill, egress to `github.com` only,
- a **searcher** with web-search, egress to DuckDuckGo only,
- a **writer** with a summarizer skill and **no egress** at all.

The skills you published to your registry in *Skills & Governance* **become the roles.** And because each agent has its own policy, the writer literally can't phone home and the reader can't reach anything but GitHub — the blast radius of any one agent is its job description.

## Spin the fleet — one command, not N setups

You don't want to repeat *Give It a Brain* + *Skills* by hand for every agent. The repo ships a helper that reads a one-line-per-agent spec and brings the whole fleet up — **create → policy → config → install skill → start gateway** — in a loop:

```text
# fleet.txt  —  name : skill : the one host this agent may reach
reader   : @workshop/repo-reader : github.com
searcher : web-search            : duckduckgo.com
writer   : -                     : -
```

```bash
./scripts/fleet.sh up fleet.txt        # bring the whole fleet up
./scripts/fleet.sh status              # watch each become Ready
./scripts/fleet.sh down                # tear it all down
```

Each line becomes a sealed agent with exactly that skill and exactly that egress rule — everything else stays deny-by-default, plus the in-cluster registry it installs from. Adding a fourth specialist is **one more line**, no extra steps.

<Callout type="note" title="Why this stays cheap to run untrusted">
Every agent is created from the same governed baseline — sealed pod, deny-by-default egress, skills only from your registry. So "add an agent" never means "widen the trust boundary": a new specialist can reach exactly one more host, and that grant is in the audit log.
</Callout>

## The orchestrator — built right into this page

No separate app to run — the conductor lives **in this website**. Type a task below and the site (1) asks the model over the **completions API** to break it into steps, (2) routes each step to the specialist whose skill fits — dispatching the real work to that sealed agent via `openclaw agent`, (3) asks the model to merge the results.

<Orchestrator />

The planning and synthesis hit `/v1/chat/completions`; each sub-task runs **inside its agent's seal** with that agent's skill and egress. The agents do the governed work; the page just conducts. Watch it live on the [Live](/learn/live) **Policy decisions** tab — each agent touching only its allowed host. (Bring the fleet up first with `./scripts/fleet.sh up fleet.txt`.)

## We design this together

`fleet.txt` + `fleet.sh` + `orchestrate.mjs` are the **starter kit**. The brief — what the fleet actually *does*, the exact roles, and how submissions are scored — we'll shape live in the session. Make it yours: swap in your own registry skills, give each agent a sharper `IDENTITY.md`, add a **reviewer** agent that critiques the writer before it ships.
