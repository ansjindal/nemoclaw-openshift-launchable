You've created one agent, given it a brain, and made it yours. The hackathon challenge is to go further: build something **useful** out of what OpenShell does best — **governed agents that do real external work**, and **more than one of them**.

## The shape of the challenge: a multi-agent app

A single sealed agent is powerful. Several, each with its own identity, policy, and job — coordinating — is a system. On this platform you can:

- **Spin up multiple sandboxes** (`openshell sandbox create`), each a specialist with a *different* egress policy — one may reach a data API, another a code host, another only inference.
- **Give each a role** via its workspace identity (`IDENTITY.md` / `SOUL.md`), so they behave differently.
- **Let them serve endpoints** (`openshell forward`) so a front agent, a tool agent, and a reviewer agent can be wired into one flow — and demoed at a URL.

What it *does* is up to you — a research-and-summarize crew, a content pipeline, a triage-and-act assistant, a "build me X" team. The constraint that makes it interesting (and safe to run untrusted) is the same one you've been using: **each agent reaches only what its policy allows.**

<Callout type="note" title="What an OpenShell agent is — and isn't — for">
These agents do <strong>governed external work</strong> (call APIs, models, services you allow) and serve results. They are deliberately <strong>sealed from the in-cluster Kubernetes API</strong> — that isolation is the product. Design your app around external tasks + multiple cooperating sandboxes, not around an agent administering the cluster it runs in.
</Callout>
