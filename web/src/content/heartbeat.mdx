So far your agent only acts when **you** talk to it. A **heartbeat** makes it act on its own: on a fixed schedule, the gateway wakes the agent with a prompt you choose, it runs one turn, and goes back to sleep. That's the difference between a chatbot and an *operator* — something that keeps watch while you're not looking.

## The idea

A heartbeat is a **cron-like self-prompt**. Every interval (default **30 minutes**), the OpenClaw gateway starts a turn for the agent with a prompt like *"Run your checklist. Reply `HEARTBEAT_OK` if all is well, or raise an alert."* The agent does its work — check a service, summarise new items, tidy its workspace — and the result is logged (and can notify you). Because it's a normal turn, it's still **fully governed**: the heartbeat agent can only reach the hosts your egress policy allows.

<Callout type="note" title="Heartbeat vs. cron">
OpenClaw also has plain <code>cron</code> jobs (run a fixed command on a schedule). A <strong>heartbeat</strong> is higher-level: it's the agent <em>thinking</em> on a schedule, with its own context and tools — not just running a script. Use cron for deterministic tasks, a heartbeat for "keep an eye on things."
</Callout>

## 1. Give it a checklist — `HEARTBEAT.md`

Just like `SOUL.md` and `IDENTITY.md`, the heartbeat reads a workspace file for *what to do each beat*. Drop a `HEARTBEAT.md` into the agent's workspace:

```bash
openshell sandbox exec -n shifty -- sh -c 'printf "%s\n" "# Heartbeat checklist" "" "Every beat, do the following and keep it short:" "1. Note the current time and that you are alive." "2. If you have pending tasks in NOTES.md, summarise their status." "3. If everything is nominal, reply with exactly: HEARTBEAT_OK" "4. If something needs my attention, start your reply with: ALERT —" > /sandbox/.openclaw/workspace/HEARTBEAT.md'
```

<Callout type="note" title="Why printf, not a heredoc">
<code>openshell sandbox exec -- sh -c '…'</code> rejects a quoted argument containing literal newlines, so a <code>cat &lt;&lt;EOF</code> heredoc fails. <code>printf "%s\n" …</code> writes the multi-line file from a <strong>single line</strong> (the <code>\n</code> are interpreted by printf, not sent as real newlines).
</Callout>

## 2. Turn the heartbeat on

Heartbeat lives under `agents.defaults.heartbeat` in `openclaw.json`. Set the interval, the wake prompt, and where it runs. We use `openclaw config set` for each key, then **restart the gateway** so it picks them up:

```bash
openshell sandbox exec -n shifty -- sh -c 'openclaw config set agents.defaults.heartbeat.every 30m; openclaw config set agents.defaults.heartbeat.prompt "Read HEARTBEAT.md and run the checklist."; openclaw config set agents.defaults.heartbeat.isolatedSession true; openclaw config set agents.defaults.heartbeat.lightContext true; openclaw config get agents.defaults.heartbeat'
```

- **`every`** — the interval (`30m`, `1h`, `15m`…). The default is `30m`.
- **`prompt`** — the wake message. Point it at your `HEARTBEAT.md`.
- **`isolatedSession`** — run each beat in its **own session** so it doesn't pollute your interactive chat history.
- **`lightContext`** — load a trimmed context (cheaper tokens) since a heartbeat usually doesn't need the full backstory.

You can also scope it to working hours with **`activeHours`** (e.g. only beat 09:00–18:00) so a demo agent isn't burning tokens overnight.

Restart the gateway to apply (same command as in *Give It a Brain*):

```bash
openshell sandbox exec -n shifty -- sh -c \
  'cd /sandbox && [ -f gateway.pid ] && kill "$(cat gateway.pid)" 2>/dev/null; sleep 2; setsid nohup openclaw gateway run --port 30789 --bind lan --auth password --password openshell-wad26 --allow-unconfigured >/sandbox/gateway.log 2>&1 < /dev/null & echo $! > gateway.pid; sleep 8; grep -E "listening on ws|heartbeat" /sandbox/gateway.log | tail -3'
```

## 3. Watch a beat happen

You don't have to wait 30 minutes — tail the gateway log and you'll see the heartbeat fire on schedule, the agent take a turn, and a `HEARTBEAT_OK` (or an `ALERT —`) come back:

```bash
openshell sandbox exec -n shifty -- sh -c 'tail -n 40 /sandbox/gateway.log | grep -iE "heartbeat|HEARTBEAT_OK|ALERT" || echo "(no beat yet — first one is one interval away)"'
```

Each beat is also an audited turn — you can see it in `openshell logs shifty` like any other agent action.

A heartbeat is what turns a sandboxed model into something that <strong>runs</strong>: it watches, checks, and reports on its own — inside the same deny-by-default seal. In Part VI you'll use this to keep a multi-agent app alive between user requests.
</Callout>
