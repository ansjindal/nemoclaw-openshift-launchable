Your sandbox is up, but it's empty — no model, no running agent. OpenClaw reads its config from `~/.openclaw/openclaw.json` (inside the sandbox, `HOME` is `/sandbox`). Let's give it a model and start the **OpenClaw gateway** (its Control UI + agent runtime).

## 1. Build the config — pointed at the privacy router

This points OpenClaw at the gateway's privacy router, `https://inference.local` — **not** your real endpoint. Your key stays in the gateway (you registered it in the inference lesson); the agent only needs the **model id** and a placeholder key. We build the config on the host, then stage it **into the sandbox** — and because the agent is Landlock-confined, we write it through `openshell sandbox exec` (base64, since exec doesn't take multi-line args):

```bash
cd ~/nemoclaw-openshift-launchable
set -a; . .env; set +a
PW="${OPENCLAW_GATEWAY_PASSWORD:-openshell-wad26}"
OC=$(jq -n --arg model "$NEMOCLAW_MODEL" --arg pw "$PW" '{
    gateway: { auth:{password:$pw}, remote:{ url:"ws://127.0.0.1:30789", password:$pw },
               controlUi:{ dangerouslyAllowHostHeaderOriginFallback:true } },
    agents:  { defaults:{ model:{ primary:("custom/"+$model) } } },
    models:  { providers:{ custom:{ baseUrl:"https://inference.local/v1",
               apiKey:"openshell-router", api:"openai-completions",
               models:[{id:$model,name:$model}] } } }
  }')
echo "$OC" | jq .   # <- see exactly what we're about to write
openshell sandbox exec -n shifty -- sh -c "mkdir -p /sandbox/.openclaw && echo $(printf '%s' "$OC" | base64 -w0) | base64 -d > /sandbox/.openclaw/openclaw.json"
openshell sandbox exec -n shifty -- openclaw config validate
```

That `echo "$OC" | jq .` prints the exact `openclaw.json` you're staging — it looks like this:

```json
{
  "gateway": {
    "auth": { "password": "openshell-wad26" },
    "remote": { "url": "ws://127.0.0.1:30789", "password": "openshell-wad26" },
    "controlUi": { "dangerouslyAllowHostHeaderOriginFallback": true }
  },
  "agents": { "defaults": { "model": { "primary": "custom/<your-model>" } } },
  "models": {
    "providers": {
      "custom": {
        "baseUrl": "https://inference.local/v1",
        "apiKey": "openshell-router",
        "api": "openai-completions",
        "models": [ { "id": "<your-model>", "name": "<your-model>" } ]
      }
    }
  }
}
```

`Config valid` means OpenClaw accepted it. Two things to notice: `models.providers.custom.baseUrl` is `inference.local` (not your real endpoint), and `apiKey` is the literal placeholder `openshell-router` — the gateway swaps in your real key when it routes the call upstream.

<Callout type="note" title="Why stage through the gateway, not oc cp">
The agent is sealed under <strong>Landlock</strong> — it can only read <code>/sandbox</code> and <code>/tmp</code>. A file you <code>oc cp</code> into <code>/workspace</code> is invisible to it. Writing through <code>openshell sandbox exec</code> lands in the agent's real view. Same isolation, everywhere.
</Callout>

## 2. Start (or restart) the OpenClaw gateway

So far the sandbox has just been **idling** — `create` only ran `true`, and the supervisor keeps the pod alive. Now we launch **OpenClaw** inside it, with the config you just wrote — on port `30789` (LAN bind so we can expose it next), password auth.

We start it under `setsid` so it runs in its **own session** and keeps running after this command returns (a plain background process would be killed when the `exec` session ends). We `pkill` any previous instance first — so this same command is also how you **restart** OpenClaw to pick up later config changes (run it again any time):

```bash
openshell sandbox exec -n shifty -- sh -c \
  'pkill -f "openclaw gateway run"; sleep 1; cd /sandbox && setsid nohup openclaw gateway run --port 30789 --bind lan --auth password --password '"${PW:-openshell-wad26}"' --allow-unconfigured >/sandbox/gateway.log 2>&1 < /dev/null & sleep 6; grep -E "listening on ws|Browser control" /sandbox/gateway.log'
```

<Callout type="note" title="Why not `openclaw gateway restart`?">
There <em>is</em> an <code>openclaw gateway restart</code> — but it manages an <strong>installed OS service</strong> (systemd/launchd). This sandbox is a minimal container with no init system, so we run the gateway in the foreground ourselves. (<code>gateway run --force</code> won't help either: it needs <code>fuser</code>/<code>lsof</code> to free the port, and neither ships in the image.) Hence <code>pkill</code> + <code>setsid nohup</code> — the portable way to restart a foreground process in here.
</Callout>

The line that matters is **`listening on ws://0.0.0.0:30789`** — that's your agent's Control UI + WebSocket gateway (the port your Brev `openclaw` link maps to), running inside the sealed sandbox.

<Callout type="note" title="What about port 30791?">
You'll also see <code>Browser control listening on http://127.0.0.1:30791</code>. That's a <em>separate</em> helper OpenClaw always starts for browser automation (the Chrome-extension relay) — bound to <strong>127.0.0.1 only</strong>, inside the sandbox, so it's never exposed. Ignore it; your gateway is the <code>:30789</code> one. The <code>security warning</code>, <code>update available</code>, and <code>Chrome extension relay</code> lines are likewise harmless notices.
</Callout>

## 3. Prove the brain works

Have the agent run one turn. We pass `--local` so it runs the agent **embedded** (a one-shot, no Control-UI session needed) — it still sends its thinking to `inference.local`:

```bash
openshell sandbox exec -n shifty -- sh -c 'cd /sandbox && openclaw agent --local --agent main -m "Reply with exactly: PONG" --session-id smoke'
```

You should get back `PONG`. If it answers, the whole chain is live: the agent reached the model **without ever holding your key**.

<Callout type="note" title="Why --local">
Without <code>--local</code>, <code>openclaw agent</code> first tries to reach a running OpenClaw gateway on its <em>default</em> port (18789) — ours is on 30789, so it prints <code>Gateway agent failed; falling back to embedded</code> and then works anyway. <code>--local</code> just runs embedded directly, so you skip that noisy detour for a one-shot test.
</Callout>

<Callout type="rh" title="The privacy boundary, made real">
That reply travelled: agent → <code>inference.local</code> → supervisor (intercepts before policy) → the OpenShell gateway's inference router → it injects your real key → your inference endpoint → back. The agent never saw the key, and your real endpoint never appeared in its egress policy. Next: open the UI in your browser.
</Callout>
