Before you run agent commands, understand the layer that *manages* the agent: **OpenShell**, the sandbox control plane. Everything here is plain `oc` against the cluster — no agent involved yet.

## The OpenShell gateway

OpenShell runs as a StatefulSet. Look at it:

```bash
oc -n openshell get statefulset,pods,svc
```

Its **Kubernetes compute driver** continuously watches for sandboxes. A healthy gateway logs it every minute:

```bash
oc -n openshell logs openshell-0 | grep -i "Listing sandboxes" | tail -3
```

`Listing sandboxes from Kubernetes` = the control plane is alive and watching.

## The sandbox API (a CRD)

A “sandbox” is a Kubernetes custom resource — `sandboxes.agents.x-k8s.io`. That's how the platform represents an agent's isolated home:

```bash
oc get crd sandboxes.agents.x-k8s.io
```

### List sandboxes

This is the command you asked about — list every sandbox across the cluster:

```bash
oc get sandboxes -A
```

If you haven't created an agent yet it's **empty** — you build your first in Part IV (an agent like `shifty` only shows here once it exists). The point is the *shape*: every agent the gateway makes shows up here as a `Sandbox` CR you can inspect with plain `oc` (`oc -n openshell describe sandbox <name>`). That's the whole API surface — a Kubernetes object.

## Policies — what a gateway sandbox may do

An OpenShell-managed sandbox policy is **one YAML file with two halves**: a *dynamic* network section that hot-reloads on a running sandbox, and *static* filesystem + process sections locked in at creation and enforced in the kernel. Click through each part:

<PolicyExplorer />

For sandboxes created through the OpenShell gateway, every section maps to a real enforcement point — nothing here is advisory:

| Layer | Protects against | Enforced by | Runtime change? |
| --- | --- | --- | --- |
| `network_policies` | egress / data exfiltration | CONNECT proxy + OPA, per-binary | ✅ hot-reload |
| `filesystem` | binary tampering, cred theft | **Landlock LSM** (kernel) | ❌ recreate |
| `process` | priv-esc, fork bombs, bad syscalls | **seccomp-BPF** + privilege drop | ❌ recreate |

### See it on your cluster

The repo ships the OpenShell policy your agent will use. Print the real file:

```bash
cat ~/nemoclaw-openshift-launchable/policies/openclaw-sandbox.yaml
```

It's short — here's the whole thing, and every line is a guardrail:

```yaml
version: 1

filesystem_policy:                       # what the agent may touch on disk
  include_workdir: true
  read_only:  [/usr, /lib, /proc, /dev/urandom, /app, /etc, /var/log]
  read_write: [/sandbox, /tmp, /dev/null]

landlock:
  compatibility: best_effort             # enforce the FS rules via the kernel (Landlock LSM)

process:
  run_as_user: sandbox                   # the agent runs unprivileged, never root
  run_as_group: sandbox

network_policies:                        # a MAP of named egress rules — deny-by-default
  openclaw:                              # OpenClaw's own hosts (registry, API, docs)
    name: openclaw
    endpoints:
      - { host: api.openclaw.ai, port: 443 }
      - { host: registry.npmjs.org, port: 443 }
    binaries:
      - { path: /usr/bin/node }          # ONLY node/npm may use this rule
      - { path: /usr/bin/npm }
```

Now read it block by block:

- **`version`** — the policy schema version. Just `1`.
- **`filesystem_policy`** — the agent's view of the disk. `read_only` covers the system dirs (so it can't tamper with binaries or configs); `read_write` is **only** its home (`/sandbox`), `/tmp`, and `/dev/null`. Anything not listed — `/workspace`, the host filesystem — simply isn't there for the agent.
- **`landlock`** — those filesystem rules aren't advisory. They're enforced by **Landlock**, a Linux-kernel security module, so even a compromised agent can't step outside them.
- **`process`** — the agent runs as the unprivileged `sandbox` user, never root (backed by seccomp-BPF + privilege drop).
- **`network_policies`** — the heart of it: a **map** of named rules, deny-by-default. Each rule pairs **endpoints** (the host + port the agent may reach) with **binaries** (which executables may use that egress). Here there's just one — `openclaw` — letting `node`/`npm` reach OpenClaw's API + the package registries (so the agent can fetch skills/packages), and nothing else.

Read it as one sentence: *"`/usr/bin/node` may open `:443` to `api.openclaw.ai` — `curl` may not, and nobody may reach anywhere else."* That **per-binary, per-host** egress is what stops a prompt-injected agent from exfiltrating data through an unexpected channel — a deny-all firewall with tiny labelled doors.

<Callout type="note" title="Where's the model endpoint?">
Notice the policy has <strong>no rule for inference</strong>. The agent reaches its model at the virtual host <code>https://inference.local</code>, which the supervisor intercepts <em>before</em> policy is evaluated and routes to the gateway (next section). So the real inference host never appears here — and a direct call to it is denied like anything else (<code>CONNECT tunnel failed, response 403</code>). The model works; the endpoint is invisible to the agent.
</Callout>

You'll see this exact policy **enforced live** in the next lesson (on a throwaway sandbox) and again when you build your agent in Part IV — `openshell policy get <name>` prints the *effective* policy. The gateway compiles this file into kernel-level (Landlock/seccomp) and proxy-level (OPA egress) rules.

<Callout type="tip" title="The iterate loop">
For a gateway-created sandbox, start in <strong>audit</strong> mode (violations logged, not blocked), watch <code>openshell logs &lt;name&gt;</code> for denials, add the endpoints you actually need to <code>network_policies</code>, then flip to <strong>enforce</strong>. Tighten without downtime — network rules hot-reload.
</Callout>

<Callout type="note" title="Who reconciles the sandbox?">
The agent-sandbox <strong>controller</strong> (installed with OpenShell in phase 30) turns the <code>Sandbox</code> CR into a real pod, under the namespace's RBAC + Security Context Constraints. OpenShell is the broader control plane / supervisor for sandboxes.
</Callout>

## Inference routing — the privacy router

The gateway is also where inference is configured, as two control-plane objects you'll use in Part IV:

- A **provider** — a named bundle the gateway stores: an API type (`openai`, `anthropic`, …), the **endpoint** (config), and the **key** (a credential). The key lives here, gateway-side — never in a sandbox.
- The **inference route** — the gateway saying "use *this provider* + *this model* for inference." One route is active at a time.

```bash-ref
openshell provider create --name default --type openai \
  --credential OPENAI_API_KEY=…  --config OPENAI_BASE_URL=…   # key held by the gateway
openshell inference set --provider default --model …          # bind it as the active route
openshell inference get                                       # confirm
openshell provider list                                       # never prints key values
```

These are shown for reference — you'll actually run them in **Part IV**, not here. An agent then sends its thinking to **`https://inference.local`**. The supervisor intercepts that virtual host *before* the egress policy runs and hands it to the gateway, which injects the real key and calls the provider upstream. Two consequences, both good: the agent **never holds your key**, and your real inference host **never needs a `network_policies` entry** (which is why the policy above has none). That's the OpenShell privacy router — you'll set it up, then watch a sandbox use it, in Part IV.
