Before you run agent commands, understand the layer that *manages* the agent: **OpenShell**, the sandbox control plane. Everything here is plain `oc` against the cluster — no agent involved yet.

## The OpenShell gateway

OpenShell runs as a StatefulSet. Look at it:

```bash
oc -n openshell get statefulset,pods,svc
```

Its **Kubernetes compute driver** continuously watches for sandboxes. A healthy gateway logs it every minute:

```bash
oc -n openshell logs openshell-0 | grep -i "Listing sandboxes" | tail -3
```

`Listing sandboxes from Kubernetes` = the control plane is alive and watching.

## The sandbox API (a CRD)

A “sandbox” is a Kubernetes custom resource — `sandboxes.agents.x-k8s.io`. That's how the platform represents an agent's isolated home:

```bash
oc get crd sandboxes.agents.x-k8s.io
```

### List sandboxes

This is the command you asked about — list every sandbox across the cluster:

```bash
oc get sandboxes -A
```

You'll see `openshell/shifty` — the agent you created. Inspect it:

```bash
oc -n openshell describe sandbox shifty | head -40
```

## Policies — what a gateway sandbox may do

An OpenShell-managed sandbox policy is **one YAML file with two halves**: a *dynamic* network section that hot-reloads on a running sandbox, and *static* filesystem + process sections locked in at creation and enforced in the kernel. Click through each part:

<PolicyExplorer />

For sandboxes created through the OpenShell gateway, every section maps to a real enforcement point — nothing here is advisory:

| Layer | Protects against | Enforced by | Runtime change? |
| --- | --- | --- | --- |
| `network_policies` | egress / data exfiltration | CONNECT proxy + OPA, per-binary | ✅ hot-reload |
| `filesystem` | binary tampering, cred theft | **Landlock LSM** (kernel) | ❌ recreate |
| `process` | priv-esc, fork bombs, bad syscalls | **seccomp-BPF** + privilege drop | ❌ recreate |

### See it on your cluster

The repo ships the OpenShell policy your agent uses (deny-by-default: inference + the OpenClaw hosts). Read the real file:

```bash
sed -n '1,60p' ~/nemoclaw-openshift-launchable/policies/openclaw-sandbox.yaml
```

Now see the **effective** policy on your live agent — what the gateway is actually enforcing:

```bash
openshell policy get shifty
```

<Callout type="tip" title="The iterate loop">
For a gateway-created sandbox, start in <strong>audit</strong> mode (violations logged, not blocked), watch <code>openshell logs &lt;name&gt;</code> for denials, add the endpoints you actually need to <code>network_policies</code>, then flip to <strong>enforce</strong>. Tighten without downtime — network rules hot-reload.
</Callout>

<Callout type="note" title="Who reconciles the sandbox?">
The agent-sandbox <strong>controller</strong> (installed with OpenShell in phase 30) turns the <code>Sandbox</code> CR into a real pod, under the namespace's RBAC + Security Context Constraints. OpenShell is the broader control plane / supervisor for sandboxes.
</Callout>
