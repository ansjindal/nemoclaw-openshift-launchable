You listed sandboxes with `oc get sandboxes -A`. Each one is reconciled into a **pod** — and that pod is where the OpenClaw agent (and its CLI) actually lives. To run *agent* commands, you go **into the sandbox**.

## A sandbox is a pod

```bash
oc -n openclaw get pod openclaw-sandbox -o wide
```

The pod runs two containers: `openclaw` (the gateway/agent) and a helper. The agent CLI ships inside the `openclaw` container.

## Run a command inside it

You reach the agent CLI with `oc exec` into that container. The CLI is `node dist/index.js`:

```bash
oc -n openclaw exec openclaw-sandbox -c openclaw -- node dist/index.js config validate
```

That ran **inside the sandbox** — which is why it works: on the gateway host the CLI uses the gateway's local owner identity (no token/password needed).

## The `openclaw` helper

Typing that prefix every time is tedious, so this lab shell defines a helper that wraps it:

```bash
type openclaw
```

It's just:

```bash
openclaw() { oc -n openclaw exec openclaw-sandbox -c openclaw -- node dist/index.js "$@"; }
```

So from here on, `openclaw <args>` means “run the agent CLI inside the sandbox.” Try the help:

```bash
openclaw --help
```

<Callout type="tip" title="Mental model">
<strong>OpenShell</strong> manages sandboxes (the control plane, via <code>oc</code>). <strong>Inside a sandbox</strong> you run the agent's own commands (via <code>oc exec</code>, wrapped as <code>openclaw …</code>). Keep the two layers straight and everything else clicks.
</Callout>

Now you're ready to read Shifty's brain, pair your browser, and add a model — all `openclaw …` commands that run inside the sandbox.
