You created `shifty` with `openshell sandbox create`. Each sandbox is reconciled into a **pod** — but a *sealed* one. To run *agent* commands you go **into the sandbox**, and there are two ways in that land in two different worlds. Understanding the difference is understanding OpenShell.

## A sandbox is a pod

```bash
oc -n openshell get pod shifty -o wide
```

It's a normal pod with an `agent` container. But the OpenClaw agent runs in a **private network namespace** the supervisor built — not the pod's main namespace.

## Two ways in

**`oc exec`** lands in the pod's *main* namespace — full filesystem, but **not** the agent's sealed network:

```bash
oc -n openshell exec shifty -c agent -- sh -c 'echo HTTPS_PROXY="$HTTPS_PROXY"; ls /sandbox >/dev/null 2>&1 && echo "can see /sandbox" || echo "no /sandbox here"'
```

**`openshell sandbox exec`** lands in the *agent's* sealed context — Landlock filesystem (`/sandbox`, `/tmp`) and the governed egress proxy. This is where the agent actually lives:

```bash
openshell sandbox exec -n shifty -- sh -c 'echo HTTPS_PROXY="$HTTPS_PROXY"; whoami; pwd'
```

Notice the agent's shell has an `HTTPS_PROXY` pointing at the supervisor's egress proxy — every outbound call is forced through the policy. The `oc exec` shell doesn't. That gap *is* the seal.

## Run the agent CLI

The `openclaw` binary lives inside the sandbox; run it through the sealed exec:

```bash
openshell sandbox exec -n shifty -- openclaw --help
openshell sandbox exec -n shifty -- openclaw config get
```

<Callout type="tip" title="Mental model">
<strong>OpenShell</strong> manages sandboxes from outside (the control plane: <code>openshell …</code> / <code>oc</code>). <strong>Inside a sandbox</strong> you run the agent's own commands with <code>openshell sandbox exec -n shifty -- openclaw …</code>. Use <code>oc exec</code> only for the pod's outer shell — it bypasses the agent's network seal, so it's not how the agent sees the world.
</Callout>

<Callout type="rh" title="Why the seal matters">
This is the same isolation that stops a gateway sandbox from reaching the in-cluster Kubernetes API: the agent only egresses through the governed proxy. You can inspect the pod from outside all you like; the agent itself is boxed in. That's what lets you run untrusted agents safely.
</Callout>
