Your agent has a chat UI and a CLI — but the most useful door is the one your **own code** can knock on. The OpenClaw gateway can expose an **OpenAI-compatible** `/v1/chat/completions` endpoint, so any tool, script, or SDK that already speaks OpenAI can talk to your *governed, sandboxed* agent with a one-line base-URL change.

## 1. Enable the endpoint

It's off by default. Turn it on in `openclaw.json` and restart the gateway:

```bash
openshell sandbox exec -n shifty -- sh -c \
  'openclaw config set gateway.http.endpoints.chatCompletions.enabled true; openclaw config get gateway.http.endpoints'
```

```bash
openshell sandbox exec -n shifty -- sh -c \
  'cd /sandbox && [ -f gateway.pid ] && kill "$(cat gateway.pid)" 2>/dev/null; sleep 1; setsid nohup openclaw gateway run --port 30789 --bind lan --auth password --password openshell-wad26 --allow-unconfigured >/sandbox/gateway.log 2>&1 < /dev/null & echo $! > gateway.pid; sleep 6; grep -E "listening on ws" /sandbox/gateway.log | tail -1'
```

The endpoint rides on the **same `:30789`** as the Control UI and WebSocket — the port you already exposed to the host with `openshell forward` in *Open the Agent UI*. So it's reachable on the host at `http://127.0.0.1:30789/v1/chat/completions`.

## 2. Call it like OpenAI

Authenticate with your **gateway password** as a bearer token, and use the model id **`openclaw`** — the gateway routes that to whatever model your agent is configured with (your inference endpoint, via the privacy router). From the lab shell:

```bash
curl -s http://127.0.0.1:30789/v1/chat/completions \
  -H "Authorization: Bearer openshell-wad26" \
  -H "Content-Type: application/json" \
  -d '{"model":"openclaw","messages":[{"role":"user","content":"Reply with exactly: PONG"}]}' \
  | jq -r '.choices[0].message.content'
```

You get back `PONG`. That round-trip went: your `curl` → the in-sandbox OpenClaw gateway → `inference.local` → the OpenShell privacy router (real key injected) → your endpoint → back — **the full sealed chain, behind an OpenAI-shaped API.**

<Callout type="note" title="Point any OpenAI client at it">
Anything that takes an OpenAI base URL works. In code: set <code>base_url="http://127.0.0.1:30789/v1"</code>, <code>api_key="openshell-wad26"</code>, <code>model="openclaw"</code>, and call <code>chat.completions.create(...)</code> as usual. Your script never sees your real inference key, and the agent stays inside its sandbox.
</Callout>

<Callout type="note" title="Why this is more than a model proxy">
This isn't a raw passthrough to your LLM — it's your <strong>agent</strong>: identity, tools (web search, skills), and policy all apply. So an OpenAI-shaped call can trigger a governed web search or a ClawHub skill, not just a bare completion.
</Callout>

<Callout type="rh" title="Your agent is now a building block">
A stable, OpenAI-compatible URL means your sandboxed agent can sit behind a web app, a CI step, or another agent — composable infrastructure, not a chat toy. That's exactly what the Part VI multi-agent build leans on.
</Callout>
