**OpenShell** (NVIDIA) is what makes "run an autonomous agent *safely*" real. The pitch in one line: **to be useful, agents need access; to be safe, they need Zero Trust** — and OpenShell is the runtime that gives you both. It's not a single process; it's a gateway plus a set of guardrails, sitting on an open-source Kubernetes primitive.

## Inside the runtime

Two paths run through OpenShell: the **agent path** (how an agent comes alive and does work) and the **guardrail path** (what bounds it). Toggle between *what each piece is* and *what it becomes on Kubernetes*, and click any component:

### The agent path
- **Gateway** — the control plane (gRPC :8080). It allocates sandboxes, mints each a scoped identity, and hosts the policy engine + inference router. On K8s it's a StatefulSet whose *compute driver* watches the Sandbox CRD.
- **Sandbox Supervisor** (`openshell-sandbox`) — sideloaded into **every** sandbox as an init-container. It builds the agent's isolated network namespace, exchanges a bootstrap token for a sandbox JWT (`IssueSandboxToken`), relays exec/SSH, and enforces sandbox-side policy.
- **Agent Harness** — the agent itself (Definition · Memory · Skills), wrapped by the supervisor in every gateway-created sandbox. Your `shifty` agent is exactly this: the OpenClaw image + a `.md` workspace, running as an `agents.x-k8s.io` `Sandbox` the gateway manages.
- **Sub-Agents** — focused agents the harness can spawn, each its own sandbox with its own identity and policy.
- **Accelerated Tooling** — what the agent reaches *through the guardrail*: filesystem, search, Omniverse/CAD, models — via API, CLI, or MCP.

### The guardrail path
- **Policy Engine + Validator** — deny-by-default. Per binary, which hosts/methods/paths are allowed; a draft validator can propose additions for a human to approve.
- **Network Guardrail** — kernel-level, per-binary egress (`git → github.com ✓`, `curl → anywhere ✗`). Stops exfil through unknown channels; instant, no restart on change.
- **Privacy Router** — gateway-created sandboxes can call one URL, `inference.local`; the router strips sandbox-local creds and injects the real backend key + model. In this workshop the agent uses a simpler **direct provider config** (`openclaw.json`) pointed at a remote endpoint — same idea, fewer moving parts.
- **Model** — where inference runs. Here, a remote OpenAI-compatible endpoint (no local GPU); in a GPU cluster, an in-cluster vLLM.

<Callout type="note" title="Why a runtime (not just a prompt)">
Safety as a system prompt is a suggestion; safety at the <em>network and process layer</em> is enforced. Even if the agent is prompt-injected, the guardrail only lets its binaries reach the hosts and paths you allowed.
</Callout>

## On Kubernetes — the real deployment

The Helm chart deploys the gateway as a **StatefulSet**; the **agent-sandbox controller** turns `Sandbox` CRs into pods; each pod's **supervisor** dials back to the gateway over mTLS. Click any object — the detail shows how *this* launchable maps to it:

### Five enforcement layers

Every sandbox is bounded at five independent layers — some locked at creation, some hot-reloadable:

| Layer | Protects against | Enforcement point | Change at runtime? |
| --- | --- | --- | --- |
| **Network** | egress / data exfiltration | CONNECT proxy + OPA policy (per-binary) | ✅ `openshell policy set` |
| **Filesystem** | binary tampering, cred theft | Landlock LSM (kernel) | ❌ recreate sandbox |
| **Process** | priv-esc, fork bombs, bad syscalls | seccomp-BPF + privilege drop | ❌ recreate sandbox |
| **Inference** | credential exposure | privacy router intercepts `inference.local` | ✅ `openshell inference set` |
| **Gateway** | unauth access, transport tampering | mTLS + token auth (cluster CA) | config |

That's the defense-in-depth for OpenShell-governed sandboxes: even a fully prompt-injected agent is boxed in by the kernel and the proxy, not by a prompt. On top, OpenShift contributes its own half of the story — SCCs, pod security settings, and NetworkPolicy — the cluster-native guardrails every sandbox inherits.

<Callout type="note" title="Where our instance simplifies">
Single-node, CPU-only, NodePort instead of Envoy/LoadBalancer, remote inference instead of a local model, and unauthenticated CLI instead of OIDC. The <em>shape</em> is identical to the production diagram — fewer optional add-ons.
</Callout>

## The foundation: agent-sandbox (the SIG project)

The bottom layer is community Kubernetes, not NVIDIA-specific: the `Sandbox` CRD (`agents.x-k8s.io`) and its controller come from **[kubernetes-sigs/agent-sandbox](https://github.com/kubernetes-sigs/agent-sandbox)**. OpenShell adds the gateway, supervisor, privacy router, and policy *on top*. That's why `oc get sandboxes -A` works with plain `oc` — they're standard Kubernetes objects.

<Callout type="note" title="Version detail that matters">
We pin agent-sandbox to <strong>v0.4.6</strong> (the last <code>v1alpha1</code>-only release). The gateway 0.0.70 verifies a sandbox pod is owned by a <code>v1alpha1</code> Sandbox; agent-sandbox v0.5.0 switched to <code>v1beta1</code> ownerReferences, which the gateway rejects — so gateway-driven sandboxes would crash-loop. Pinning v0.4.6 is what makes <code>openshell sandbox create</code> reach <code>Ready</code> on MicroShift.
</Callout>

## How you reach the gateway

Inside the cluster the gateway is a Service, `svc/openshell:8080`. Your workshop shell runs on the **VM, outside the cluster**, so the Helm chart exposes that Service as a **NodePort** — `30808` on the MicroShift node (which itself lives *inside* the podman container, so the NodePort isn't on the VM yet). A tiny `socat` systemd unit then forwards the VM's `:30808` to the node's `10.x.x.x:30808`, so the CLI — registered against `http://127.0.0.1:30808` — reaches the gateway:

```bash
openshell status
```

<Callout type="note" title="Why it says http://127.0.0.1:30808">
That endpoint is local <em>to the VM</em> — not because the gateway is local, but because the host forwarder bridges <code>127.0.0.1:30808</code> → the in-cluster NodePort → <code>svc/openshell</code>. Production swaps this NodePort + forwarder for an ingress + hostname (Gateway API / Envoy); the <code>openshell</code> commands are identical either way.
</Callout>

## Drive the gateway from the CLI

`openshell status` is just the start. The CLI is how you'll do everything — see the verbs:

```bash
openshell --help
```

Run a full preflight (it checks the gateway, the compute driver, and the CRD are all healthy):

```bash
openshell doctor check
```

See which gateways the CLI knows about — you're pointed at the one we registered, `cluster`:

```bash
openshell gateway list
```

And list the agents running on it:

```bash
openshell sandbox list
```

If you haven't built an agent yet, this is **empty** — you create your first one in **Part IV**. (If you do see one like `shifty`, it's simply been created already.) Part III (next) goes deeper on the control plane and the policy model, with a throwaway sandbox to play on, so you understand every piece before you build for real.

Because the gateway is just gRPC, *anything* that speaks it can drive the fleet — the `openshell` CLI you just used, the companion `openshell-ui` web console, or live panels wired straight into this workshop site from the gateway API.

<Callout type="note" title="Two things both called &quot;gateway&quot;">
You'll hear "gateway" twice: the <strong>OpenShell gateway</strong> (control plane, :30808 → :8080) and the <strong>OpenClaw gateway</strong> (the agent's own control UI, :30789). Different things — keep them straight.
</Callout>
