Want to watch your **agent fleet** — CPU, memory, restarts, traffic — as you spin up more sandboxes? Add a full observability stack. This is for *you*, the operator: dashboards over the cluster and its agents.

This phase is **opt-in** (it adds real load): enable it and re-run, or run the phase directly.

## What you get

All in a `monitoring` namespace:

- **Grafana** — dashboards, on NodePort `30030` (login `admin` / `openclaw`).
- **Prometheus + Alertmanager** — metrics, 2-day retention, with `kube-state-metrics` + `node-exporter`.
- **Loki** — logs (single-binary, filesystem).
- **Tempo** — traces (OTLP `4317`/`4318`).

`kube-prometheus-stack` brings Prometheus, Alertmanager, Grafana, node-exporter and kube-state-metrics; **Loki** and **Tempo** are auto-wired into Grafana as datasources. A custom **Agent Fleet** dashboard tracks your sandbox pods' CPU, memory, and restarts.

## Turn it on

```bash
# in the repo on the instance
echo 'DEPLOY_MONITORING=true' >> .env
./scripts/80-monitoring.sh
```

The phase grants the OpenShift SCCs node-exporter needs, installs the three Helm releases, and exposes Grafana on **NodePort 30030** (already published to the host by the port-forwarder). Open it:

```bash
oc -n monitoring get pods
```

Then browse the host's port **30030** (expose it as a Brev tunnel) and log in as `admin` / `openclaw`. Look for the **OpenShell Agent Fleet** dashboard plus the built-in Kubernetes dashboards.

<Callout type="warn" title="It's not free">
Prometheus + Loki + Tempo add CPU and memory pressure on a single 4-vCPU node. Leave it off for the core workshop; turn it on when you want the fleet dashboards.
</Callout>

## Watch the fleet

Open the **OpenShell Agent Fleet** dashboard in Grafana. As you create more sandboxes (Part V's multi-agent app), each shows up here — CPU, memory, restarts per agent pod — plus the gateway's own health. This is your operator view of the whole agent fleet on one screen.

<Callout type="note" title="Why the agents can't read Prometheus themselves">
Prometheus lives <em>in-cluster</em> (a <code>10.43.*</code> ClusterIP). A gateway sandbox is network-sealed and only egresses to the <strong>external</strong> hosts its policy allows — it can't reach in-cluster services (that's the same isolation that keeps it off the Kubernetes API). Observability here is for <em>you</em>; if you want an agent to reason over metrics, expose a metrics API <em>externally</em> and allow that host in its policy.
</Callout>
