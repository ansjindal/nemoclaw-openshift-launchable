When your Brev instance booted, the launchable pre-built everything **up to and including a running OpenShift-compatible cluster**, so you can spend your time on the agent, not on `kubeadm`. This lesson shows *how* that base was made — and lets you poke at it live.

## What's underneath

The cluster is **MicroShift** — Red Hat's edge-sized OpenShift — running **inside a podman container** (a setup often called MINC, "MicroShift-in-Container"). One VM, one container, one node, but real OpenShift APIs: `Routes`, `SecurityContextConstraints`, the works.

```text
Brev VM (Ubuntu)
└── podman
    └── microshift container  ← the "node"
        └── MicroShift (single-node OpenShift)
```

The provisioning (you don't run these — they already ran) was, in essence:

```bash
# host deps: podman + tooling      (scripts/10-host-deps.sh)
# CONTAINER_ENGINE=podman — docker breaks MINC's nested mounts
# start MicroShift inside podman    (scripts/20-microshift.sh)
# → writes a kubeconfig you can use with `oc`
```

## Inspect it live

Your lab shell already has `oc` pointed at the cluster. Confirm the node is up:

```bash
oc get nodes
```

One `Ready` node = your whole cluster. Now look at what's already deployed for you:

```bash
oc get ns
oc -n openshell get pods
```

You'll see the `openshell` namespace with the **gateway** (`openshell-0`) running — that's the agent control plane the rest of the workshop uses. You might also see an agent pod like `shifty` if one's already been created; if not, *you* create yours hands-on in **Part IV**.

It really is OpenShift — check an OpenShift-only API:

```bash
oc get scc --no-headers | head -5
oc api-resources | grep -iE "route.openshift.io|security.openshift.io" | head
```

`SecurityContextConstraints` and `route.openshift.io` are OpenShift extensions a vanilla Kubernetes cluster doesn't have. That's the Red Hat layer your agents run on.

<Callout type="tip" title="Why a container, not a VM-per-node">
MINC makes the whole cluster a single disposable container — fast to boot on a Brev instance, and it still exercises the real OpenShift admission + SCC path your agents must satisfy. The trade-off (no cloud LoadBalancer) is why we reach in-cluster services through a host forwarder + NodePort — the next lesson shows how that works for the gateway.
</Callout>
