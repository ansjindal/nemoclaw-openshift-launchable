Web search was *one* tool. **Skills** (OpenClaw plugins) are how you give your agent *many* — a weather lookup, a code helper, a domain workflow — installed from a registry or written by you. The interesting part on this platform isn't just *adding* skills; it's that every skill stays **governed**: where it comes from, whether it runs, and what it can reach are all things you control.

This lesson builds to the production pattern: skills enter through **one registry you run in the cluster** — either **authored** fresh or **vetted-and-mirrored** from ClawHub — and the agent's *only* egress for installs is that registry. ClawHub and public npm are denied at the seal.

## 1. Discover skills on ClawHub

Your agent ships **locked to your private registry** — by default it can't reach ClawHub at all (you'll confirm that in step 7). But to understand *why* that's the right posture, it helps to first meet the alternative — **ClawHub**, OpenClaw's public plugin registry — and see how you'd govern pulling skills straight from it. So open a **temporary door** for the next few steps:

```bash
openshell policy update shifty --add-endpoint 'clawhub.ai:443:full' --add-endpoint 'hub.openclaw.ai:443:full' --binary /usr/bin/node --binary /usr/local/bin/node
```

Now search ClawHub from inside your sandbox:

```bash
openshell sandbox exec -n shifty -- openclaw plugins search hello
```

You'll get a short catalog, each line ending with its install command — e.g. `openclaw plugins install clawhub:openclaw-hello-world`. (Try `… plugins search weather` too.)

<Callout type="note" title="That search only worked because you opened the door">
ClawHub lives at <code>clawhub.ai</code> + <code>hub.openclaw.ai</code>. Your sandbox is deny-by-default and ships with <strong>neither</strong> — the search worked only because you just added them. That's the whole point: ClawHub is reachable <em>only</em> when your policy says so, and by the end of this lesson you'll take that door away for good, leaving your own registry as the single source.
</Callout>

## 2. Install a skill — and vet which ones the policy allows

ClawHub is an allowed *source*: the `openclaw` rule in your sandbox policy permits its registry (`clawhub.ai`) and package CDN (`hub.openclaw.ai`), for `node`/`npm` only.

```yaml
network_policies:
  openclaw:
    endpoints:
      - { host: clawhub.ai, port: 443, access: full }       # ClawHub registry
      - { host: hub.openclaw.ai, port: 443, access: full }  # …its package CDN
      # npm — note protocol: rest + allow_encoded_slash (see the callout below)
      - { host: registry.npmjs.org, port: 443, access: full, protocol: rest, allow_encoded_slash: true }
    binaries:
      - { path: /usr/bin/node }
      - { path: /usr/local/bin/node }
      - { path: /usr/bin/npm }
      - { path: /usr/local/bin/npm }
```

<Callout type="note" title="Why registry.npmjs.org needs allow_encoded_slash">
A plugin can pull <strong>scoped</strong> npm deps (e.g. <code>openclaw-hello-world</code> → <code>@sinclair/typebox</code>), which npm fetches as <code>/@sinclair%2ftypebox</code> — the <code>/</code> URL-encoded to <code>%2F</code>. The supervisor's egress proxy does strict path parsing and <strong>rejects encoded slashes by default</strong> (a path-traversal / request-smuggling guard), so the host is allowed but the request is denied by <code>engine:http-parser</code> and the install hangs at <em>"Installing plugin dependencies"</em>. Setting <code>protocol: rest</code> + <code>allow_encoded_slash: true</code> on the npm endpoint permits it. You'll see this exact denial — and then the fix — on the <a href="/learn/live">Live OpenShell</a> <strong>Policy decisions</strong> tab. (It's <strong>not</strong> settable via <code>openshell policy update --add-endpoint</code>; it lives in the policy YAML.)
</Callout>

But `access: full` lets the agent install **any** ClawHub package. OpenShell's egress proxy can do better — it terminates TLS and evaluates the **HTTP path** (`engine:l7`), so you can vet ClawHub **package-by-package**. Switch the registry to enforced mode and block one you don't trust:

```bash
openshell policy update shifty --remove-endpoint 'clawhub.ai:443'
openshell policy update shifty --add-endpoint 'clawhub.ai:443:read-only:rest:enforce' --binary /usr/bin/node --binary /usr/local/bin/node
openshell policy update shifty --add-deny 'clawhub.ai:443:GET:/api/v1/packages/weather-api-hub'
openshell policy update shifty --add-deny 'clawhub.ai:443:GET:/api/v1/packages/weather-api-hub/**'
```

Now try to install that package — it's refused at layer 7, before any code lands:

```bash
openshell sandbox exec -n shifty -- openclaw plugins install clawhub:weather-api-hub
```

```text
ClawHub /api/v1/packages/weather-api-hub failed (403): {
  "error": "policy_denied",
  "detail": "GET /api/v1/packages/weather-api-hub blocked by deny rule",
  "host": "clawhub.ai", "layer": "l7", "policy": "allow_clawhub_ai_443"
}
```

The audit records the decision:

```bash
openshell logs shifty --source sandbox | grep weather-api-hub
```

```text
HTTP:GET  DENIED GET .../packages/weather-api-hub [policy:allow_clawhub_ai_443 engine:l7]
          [reason: blocked by deny rule]
```

A package you **haven't** blocked still installs. Pull the vetted hello-world skill:

```bash
openshell sandbox exec -n shifty -- openclaw plugins install clawhub:openclaw-hello-world
```

It downloads from ClawHub and registers. Confirm it's there:

```bash
openshell sandbox exec -n shifty -- openclaw plugins list
```

You'll see **Hello World** — `enabled`, version `0.1.0`, "Registers a `hello` tool and ships a matching skill." So the agent installs from ClawHub — but only the packages you let through.

<Callout type="note" title="Three CLI things you'll hit">
<strong>Loading is deferred:</strong> <code>install</code> / <code>enable</code> / <code>disable</code> only change files + config — they take effect when the gateway <em>restarts</em>. This lesson installs several skills and restarts <strong>once</strong> at the end (step 8), so <code>plugins list</code> showing <code>enabled</code> means "configured", not "live yet".<br/>
<strong>Filtering the list:</strong> <code>plugins list</code> includes every stock provider. Narrow it with <code>--enabled</code>, or <code>--json</code> piped to <code>grep</code>/<code>jq</code> — or just <code>… plugins list | grep -i hello</code>.<br/>
<strong>Noise:</strong> the <code>[UNDICI-EHPA]</code> lines are harmless Node experimental warnings — prefix any command with <code>NODE_NO_WARNINGS=1</code> to silence them (omitted below for brevity).
</Callout>

<Callout type="note" title="Blocklist, not a one-line allowlist">
This is a <strong>blocklist</strong>: deny the packages you don't trust. OpenShell's L7 model is default-allow once a host is granted, and <strong>deny rules win over allow rules</strong> — so you can't write "allow only package X" in a single rule (a <code>deny /packages/**</code> would also kill X). The practical recipe: allow ClawHub as a source, <code>--add-deny</code> the packages you reject, and — next — enable only the skills you bless.
</Callout>

## 3. Governance, layer one — every install is on the record

Allowed or denied, each install is an audited OCSF event. Pull the trail for both packages from step 2:

```bash
openshell logs shifty --source sandbox | grep -iE 'packages/(weather-api-hub|openclaw-hello-world)'
```

```text
HTTP:GET  DENIED  .../packages/weather-api-hub                 [policy:allow_clawhub_ai_443 engine:l7]
HTTP:GET  ALLOWED .../packages/openclaw-hello-world/download   [policy:allow_clawhub_ai_443 engine:l7]
```

Both decisions are fully attributable — the rule that decided (`allow_clawhub_ai_443`), the exact package URL, the binary (`/usr/bin/node`), the layer (`l7`). In production these stream to your SIEM: every skill install, allowed or blocked, logged and traceable to the policy that governed it. That's the answer to *"can I prove which skills my agents pulled — and which I refused?"* — yes, down to the package.

<Callout type="note" title="Seeing nothing from that grep?">
Those <code>packages/…</code> path lines only appear when ClawHub is in <strong>L7 enforce</strong> mode (the <code>:rest:enforce</code> endpoint from step 2) — that's what lets the proxy read the HTTP path. If ClawHub is allowed as a plain <code>:full</code> tunnel (L4), the proxy can't see the path, so there's nothing to match; grep <code>clawhub.ai</code> instead and you'll see the <code>NET:OPEN</code> ALLOW/DENY line. And if you're already locked down (step 7), ClawHub is denied outright — you'll see <code>NET:OPEN … DENIED clawhub.ai:443</code>.
</Callout>

## 4. Governance, layer two — *which skills run* (OpenClaw allowlist)

Installed ≠ unconditionally on. OpenClaw keeps an explicit per-skill switch in `plugins.entries`:

```bash
openshell sandbox exec -n shifty -- openclaw config get plugins
```

Each entry has an `enabled` flag. Flip one off and on:

```bash
openshell sandbox exec -n shifty -- sh -c \
  'openclaw plugins disable hello-world; openclaw config get plugins.entries.hello-world; openclaw plugins enable hello-world'
```

`disable` sets `{"enabled": false}` (it takes effect on the next gateway restart). So you can ship an agent with a curated set: install many, **enable only the ones you bless**.

<Callout type="note" title="And layer three: what a skill can DO">
Even an <em>enabled</em> skill is still inside the seal. A skill that tries to reach an un-allowed host, or run a binary not in the policy's <code>binaries</code> list, is blocked and audited — exactly like any other agent action. A rogue skill can't exfiltrate or phone home unless your policy already permits that destination. <strong>Source-allowlist + enable-allowlist + egress/binary policy</strong> — three independent gates.
</Callout>

<Callout type="note" title="If a skill won't load">
Community plugins must resolve their own runtime dependencies. If one fails on restart (<code>Cannot find module …</code>), it's dropped from the registry — diagnose with <code>openclaw plugins inspect &lt;id&gt; --runtime</code>. Prefer skills that bundle their deps.
</Callout>

## 5. Author your own skill

A skill is a small **TypeScript plugin** built against the **OpenClaw plugin SDK**. That SDK ships *inside the sandbox image* — it isn't a public npm package — so the sandbox **is** your skill dev environment. You author and build here, then (next section) **publish to your registry** so the locked agent installs it like any other skill. In production this exact build runs in **CI from the same image**; the steps below are identical.

<Callout type="note" title="Why not author on your laptop?">
Because the toolchain lives in the image: <code>openclaw/plugin-sdk</code> (and the <code>openclaw</code> CLI) are only present in the sandbox image, not on npmjs. So you build <em>where the SDK is</em> — the sandbox here, or a container from the same image in CI — and the registry is what carries the result out to the agent. The sandbox is the dev box; the agent is the locked runtime; the registry is the bridge.
</Callout>

Scaffold a project:

```bash
openshell sandbox exec -n shifty -- openclaw plugins init my-skill --name "My Skill" --directory /sandbox/myskill
```

That writes `openclaw.plugin.json` (the manifest), `src/index.ts` (your tool code), `package.json`, and a test. Edit `src/index.ts` to register your tool, then build + package it:

```bash
openshell sandbox exec -n shifty -- sh -c 'cd /sandbox/myskill && npm install && npm run plugin:build'
```

`npm run plugin:build` runs `tsc` then `openclaw plugins build` → emits `dist/index.js` and finalizes the manifest. Install your built skill:

```bash
openshell sandbox exec -n shifty -- openclaw plugins install /sandbox/myskill
```

You'll see **`Installed plugin: my-skill`** — its `openclaw` peer dependency is linked from the runtime, so you don't vendor it. Restart the gateway to load it (the same restart from *Give It a Brain*), then it appears in `plugins list`, `enabled`, governed by exactly the same policy (registry + enablement + egress) as any ClawHub skill.

Installing from a local folder is fine for *testing*. For real use you want skills to come from a **registry you control** — versioned, shared across agents, lockable. That's the rest of this lesson.

## 6. Publish to a registry you run in the cluster

We run a private **[Verdaccio](https://verdaccio.org) npm registry** *inside the cluster* — an ordinary Deployment + Service in the `openshell` namespace, reachable at `registry.openshell.svc.cluster.local:4873`. Skills are just npm packages, so a standard npm registry is all you need.

### How does `install @workshop/regtest` know to use it?

There's no OpenClaw-specific "registry endpoint" setting — **it's plain npm scope routing.** `openclaw plugins install @workshop/regtest` is an *npm spec*, so OpenClaw hands resolution to `npm`, and npm routes by **scope** using your sandbox's `~/.npmrc` (`/sandbox/.npmrc`):

```ini
@workshop:registry=http://registry.openshell.svc.cluster.local:4873/
//registry.openshell.svc.cluster.local:4873/:_auth=${TOKEN}   # workshop publish/read token
```

That `@workshop:registry=…` line **is** the endpoint. Anything matching `@workshop/*` resolves from your in-cluster registry; everything else still uses npm's default `registry` (we lock that down in step 7). Nothing is special about the word `@workshop` — it's an npm scope, and you could add `@acme:registry=…` for a second source the same way.

The endpoint must also be allowed by the sandbox **egress policy** — one more rule, same shape as the ClawHub/npm ones:

```yaml
network_policies:
  in-cluster-registry:
    endpoints:
      - { host: registry.openshell.svc.cluster.local, port: 4873, protocol: rest, access: full, allow_encoded_slash: true }
    binaries:
      - { path: /usr/bin/npm }
      - { path: /usr/local/bin/npm }
      - { path: /usr/bin/node }
      - { path: /usr/local/bin/node }
```

<Callout type="note" title="In-cluster, but still proxied and audited">
The sandbox reaches the registry over the <strong>cluster network</strong> — CoreDNS resolves <code>…svc.cluster.local</code>, and the request still passes through the OpenShell egress proxy. So the install is policy-gated and lands on the <a href="/learn/live">Live</a> <strong>Policy decisions</strong> tab as <code>policy:in-cluster-registry</code>, exactly like any internet egress. Scoped names arrive URL-encoded (<code>@workshop%2Fregtest</code>) — which is why this endpoint also needs <code>allow_encoded_slash</code>.
</Callout>

### Authenticate the sandbox to your registry

Verdaccio allows **anonymous read** but **authenticated publish** (`publish: $authenticated`). So before you can publish, create a publish user and write the `/sandbox/.npmrc` shown above for real. The user is created via the registry's own API **from inside the sandbox** (the host can't reach a `…svc.cluster.local` ClusterIP — the sandbox can, through its egress proxy). Run this once in the lab shell:

```bash
openshell sandbox exec -n shifty -- sh -c '
  R=registry.openshell.svc.cluster.local:4873
  # create the publish user (Verdaccio htpasswd) — 201 = created, 409 = already exists
  node -e "const h=require(\"http\"),b=JSON.stringify({name:\"workshop\",password:\"wad26-skills\"});const q=h.request(\"http://$R/-/user/org.couchdb.user:workshop\",{method:\"PUT\",headers:{\"content-type\":\"application/json\",\"content-length\":Buffer.byteLength(b)}},r=>r.on(\"end\",()=>console.log(\"registry user:\",r.statusCode)));q.write(b);q.end();"
  # route npm at the registry (default + @workshop scope) with the publish token
  printf "registry=http://%s/\n@workshop:registry=http://%s/\n//%s/:_auth=%s\n" "$R" "$R" "$R" "$(printf workshop:wad26-skills | base64)" > /sandbox/.npmrc
  echo "wrote /sandbox/.npmrc"
'
```

That makes the `/sandbox/.npmrc` above real: `@workshop/*` (and the default registry) now resolve to Verdaccio, and the `_auth` token lets you publish. (The scripted provisioner does this same step at agent-creation time; on the hands-on path you do it here.)

### Flow A — publish your own skill

You built and tested a skill in step 5. Give it a scoped name and publish it (the sandbox is now authenticated from the step above):

```bash
openshell sandbox exec -n shifty -- sh -c 'cd /sandbox/myskill && npm pkg set name=@workshop/myskill && npm pkg delete private && npm publish'
```

`npm publish` sees the `@workshop` scope and pushes to your in-cluster registry. Now install it by name — the way any agent would:

```bash
openshell sandbox exec -n shifty -- openclaw plugins install @workshop/myskill --force
```

You'll see the scoped fetch (`GET …:4873/@workshop%2Fmyskill`, allowed under `policy:in-cluster-registry`) and then **`Installed plugin: myskill`**.

### Flow B — mirror a *vetted* ClawHub skill

The other way a skill enters your registry: take one from ClawHub, **review the code**, and republish it under your own scope. That's the supply-chain control — a human vets it *once*, and from then on agents install *your copy*, never the internet's. Take the `hello-world` you pulled in step 2, rescope it, publish:

```bash
openshell sandbox exec -n shifty -- sh -c 'cp -r /sandbox/.openclaw/extensions/hello-world /sandbox/mirror-hello && cd /sandbox/mirror-hello && npm pkg set name=@workshop/hello-world && npm publish'
openshell sandbox exec -n shifty -- openclaw plugins install @workshop/hello-world --force
```

It's now a first-class package in *your* registry (`@workshop/hello-world@0.1.0`), installed from there — no ClawHub round-trip at install time. In production this "vet → mirror → publish" is a CI step; here you did it by hand to see the mechanics.

## 7. Lock the door again — back to the default

Everything from step 1 on only worked because you **opened ClawHub temporarily**. Now close it and return to the agent's shipped posture: **the registry is its only skill source.** npm's *default* registry already points at Verdaccio (it uplinks to npmjs and caches deps), so you just drop ClawHub and npm from the policy, leaving **one egress endpoint** — this is exactly the policy your agent boots with:

```yaml
# locked policy — the only place the agent may fetch skills OR deps
network_policies:
  in-cluster-registry:
    endpoints:
      - { host: registry.openshell.svc.cluster.local, port: 4873, protocol: rest, access: full, allow_encoded_slash: true }
    binaries: [ {path: /usr/bin/npm}, {path: /usr/local/bin/npm}, {path: /usr/bin/node}, {path: /usr/local/bin/node} ]
  # clawhub.ai, hub.openclaw.ai, registry.npmjs.org — removed
```

```bash
openshell policy set shifty --policy locked.yaml --wait
```

Now prove the boundary. ClawHub is gone:

```bash
openshell sandbox exec -n shifty -- openclaw plugins install clawhub:openclaw-hello-world
```

```text
Resolving clawhub:openclaw-hello-world…
fetch failed | Proxy response (403) … | UND_ERR_ABORTED
```

```text
NET:OPEN [MED] DENIED /usr/bin/node -> clawhub.ai:443
        [policy:- engine:opa] [reason: endpoint clawhub.ai:443 is not allowed by any policy]
```

…but your registry still serves everything — skills **and** their transitive deps (Verdaccio fetches an uncached package from its npmjs uplink, caches it, serves the sandbox):

```bash
openshell sandbox exec -n shifty -- openclaw plugins install @workshop/hello-world --force   # ✓ installed
```

That denial shows in red on the Live **Policy decisions** tab. The agent now has exactly one way to gain a capability: a package *you* placed in *your* registry.

## 8. Watch the agent use it

A plugin only goes live when the gateway **restarts** (step 4 above). It's the same one-liner you used in [Give It a Brain](/learn/configure) — kill the recorded PID, relaunch under `setsid` in its own session — with a final check that `hello` loaded:

```bash
openshell sandbox exec -n shifty -- sh -c \
  'cd /sandbox && [ -f gateway.pid ] && kill "$(cat gateway.pid)" 2>/dev/null; sleep 2; setsid nohup openclaw gateway run --port 30789 --bind lan --auth password --password openshell-wad26 --allow-unconfigured >/sandbox/gateway.log 2>&1 < /dev/null & echo $! > gateway.pid; sleep 8; openclaw plugins list --enabled | grep -i hello'
```

You'll see **Hello World** in the enabled list. (Restarting the gateway briefly drops the host UI forward — it's a `Restart=always` service, so give it a few seconds to reconnect; if the UI still won't load, `sudo systemctl restart openclaw-forward.service`.)

Now open the [OpenClaw UI](/learn/live) and ask the agent, in plain English:

> use the hello tool to greet WeAreDevelopers

The agent picks the `hello` tool, calls it with `name: "WeAreDevelopers"`, and the tool returns **`Hello, WeAreDevelopers!`** — which the agent relays back in chat. (No name? It defaults to `Hello, world!`.) That's the whole point: you went from *installed* to the **model actually invoking your skill** — and it ran inside the same seal as everything else. If the skill tried to reach a host outside the policy, it'd be blocked and audited, exactly like any other action.

<Callout type="rh" title="Skills are capabilities you can prove">
On most agent stacks, "install a plugin" means handing arbitrary code your whole network. Here, a skill is gated three ways — registry, enablement, and egress — and every byte it moves is in the audit log. That's the difference between an agent that's <em>extensible</em> and one that's extensible <em>and</em> governable.
</Callout>
