Web search was *one* tool. **Skills** (OpenClaw plugins) are how you give your agent *many* — a weather lookup, a code helper, a domain workflow — installed from a registry or written by you. The interesting part on this platform isn't just *adding* skills; it's that every skill stays **governed**: where it comes from, whether it runs, and what it can reach are all things you control.

## 1. Discover skills on ClawHub

**ClawHub** is OpenClaw's plugin registry. Search it from inside your sandbox:

```bash
openshell sandbox exec -n shifty -- openclaw plugins search hello
```

You'll get a short catalog, each line ending with its install command — e.g. `openclaw plugins install clawhub:openclaw-hello-world`. (Try `… plugins search weather` too.)

<Callout type="note" title="That search already crossed a governed boundary">
ClawHub lives at <code>clawhub.ai</code> + <code>hub.openclaw.ai</code>. Your sandbox is deny-by-default, so that search only worked because those hosts are in the <code>openclaw</code> rule of your policy. Allowing exactly those two hosts is how you say <em>"skills may come from ClawHub — and nowhere else."</em>
</Callout>

## 2. Install a skill — and vet which ones the policy allows

ClawHub is an allowed *source*: the `openclaw` rule in your sandbox policy permits its registry (`clawhub.ai`) and package CDN (`hub.openclaw.ai`), for `node`/`npm` only.

```yaml
network_policies:
  openclaw:
    endpoints:
      - { host: clawhub.ai, port: 443, access: full }       # ClawHub registry
      - { host: hub.openclaw.ai, port: 443, access: full }  # …its package CDN
      # npm — note protocol: rest + allow_encoded_slash (see the callout below)
      - { host: registry.npmjs.org, port: 443, access: full, protocol: rest, allow_encoded_slash: true }
    binaries:
      - { path: /usr/bin/node }
      - { path: /usr/local/bin/node }
      - { path: /usr/bin/npm }
      - { path: /usr/local/bin/npm }
```

<Callout type="note" title="Why registry.npmjs.org needs allow_encoded_slash">
A plugin can pull <strong>scoped</strong> npm deps (e.g. <code>openclaw-hello-world</code> → <code>@sinclair/typebox</code>), which npm fetches as <code>/@sinclair%2ftypebox</code> — the <code>/</code> URL-encoded to <code>%2F</code>. The supervisor's egress proxy does strict path parsing and <strong>rejects encoded slashes by default</strong> (a path-traversal / request-smuggling guard), so the host is allowed but the request is denied by <code>engine:http-parser</code> and the install hangs at <em>"Installing plugin dependencies"</em>. Setting <code>protocol: rest</code> + <code>allow_encoded_slash: true</code> on the npm endpoint permits it. You'll see this exact denial — and then the fix — on the <a href="/learn/live">Live OpenShell</a> <strong>Policy decisions</strong> tab. (It's <strong>not</strong> settable via <code>openshell policy update --add-endpoint</code>; it lives in the policy YAML.)
</Callout>

But `access: full` lets the agent install **any** ClawHub package. OpenShell's egress proxy can do better — it terminates TLS and evaluates the **HTTP path** (`engine:l7`), so you can vet ClawHub **package-by-package**. Switch the registry to enforced mode and block one you don't trust:

```bash
openshell policy update shifty --remove-endpoint 'clawhub.ai:443'
openshell policy update shifty --add-endpoint 'clawhub.ai:443:read-only:rest:enforce' --binary /usr/bin/node --binary /usr/local/bin/node
openshell policy update shifty --add-deny 'clawhub.ai:443:GET:/api/v1/packages/weather-api-hub'
openshell policy update shifty --add-deny 'clawhub.ai:443:GET:/api/v1/packages/weather-api-hub/**'
```

Now try to install that package — it's refused at layer 7, before any code lands:

```bash
openshell sandbox exec -n shifty -- openclaw plugins install clawhub:weather-api-hub
```

```text
ClawHub /api/v1/packages/weather-api-hub failed (403): {
  "error": "policy_denied",
  "detail": "GET /api/v1/packages/weather-api-hub blocked by deny rule",
  "host": "clawhub.ai", "layer": "l7", "policy": "allow_clawhub_ai_443"
}
```

The audit records the decision:

```bash
openshell logs shifty --source sandbox | grep weather-api-hub
```

```text
HTTP:GET  DENIED GET .../packages/weather-api-hub [policy:allow_clawhub_ai_443 engine:l7]
          [reason: blocked by deny rule]
```

A package you **haven't** blocked still installs. Pull the vetted hello-world skill:

```bash
openshell sandbox exec -n shifty -- openclaw plugins install clawhub:openclaw-hello-world
```

It downloads from ClawHub and registers. Confirm it's there:

```bash
openshell sandbox exec -n shifty -- openclaw plugins list
```

You'll see **Hello World** — `enabled`, version `0.1.0`, "Registers a `hello` tool and ships a matching skill." So the agent installs from ClawHub — but only the packages you let through.

<Callout type="note" title="Blocklist, not a one-line allowlist">
This is a <strong>blocklist</strong>: deny the packages you don't trust. OpenShell's L7 model is default-allow once a host is granted, and <strong>deny rules win over allow rules</strong> — so you can't write "allow only package X" in a single rule (a <code>deny /packages/**</code> would also kill X). The practical recipe: allow ClawHub as a source, <code>--add-deny</code> the packages you reject, and — next — enable only the skills you bless.
</Callout>

## 3. Governance, layer one — every install is on the record

Allowed or denied, each install is an audited OCSF event. Pull the trail for both packages from step 2:

```bash
openshell logs shifty --source sandbox | grep -iE 'packages/(weather-api-hub|openclaw-hello-world)'
```

```text
HTTP:GET  DENIED  .../packages/weather-api-hub                 [policy:allow_clawhub_ai_443 engine:l7]
HTTP:GET  ALLOWED .../packages/openclaw-hello-world/download   [policy:allow_clawhub_ai_443 engine:l7]
```

Both decisions are fully attributable — the rule that decided (`allow_clawhub_ai_443`), the exact package URL, the binary (`/usr/bin/node`), the layer (`l7`). In production these stream to your SIEM: every skill install, allowed or blocked, logged and traceable to the policy that governed it. That's the answer to *"can I prove which skills my agents pulled — and which I refused?"* — yes, down to the package.

## 4. Governance, layer two — *which skills run* (OpenClaw allowlist)

Installed ≠ unconditionally on. OpenClaw keeps an explicit per-skill switch in `plugins.entries`:

```bash
openshell sandbox exec -n shifty -- openclaw config get plugins
```

Each entry has an `enabled` flag. Flip one off and on:

```bash
openshell sandbox exec -n shifty -- sh -c \
  'openclaw plugins disable hello-world; openclaw config get plugins.entries.hello-world; openclaw plugins enable hello-world'
```

`disable` sets `{"enabled": false}` (it takes effect on the next gateway restart). So you can ship an agent with a curated set: install many, **enable only the ones you bless**.

<Callout type="note" title="And layer three: what a skill can DO">
Even an <em>enabled</em> skill is still inside the seal. A skill that tries to reach an un-allowed host, or run a binary not in the policy's <code>binaries</code> list, is blocked and audited — exactly like any other agent action. A rogue skill can't exfiltrate or phone home unless your policy already permits that destination. <strong>Source-allowlist + enable-allowlist + egress/binary policy</strong> — three independent gates.
</Callout>

<Callout type="note" title="If a skill won't load">
Community plugins must resolve their own runtime dependencies. If one fails on restart (<code>Cannot find module …</code>), it's dropped from the registry — diagnose with <code>openclaw plugins inspect &lt;id&gt; --runtime</code>. Prefer skills that bundle their deps.
</Callout>

## 5. Author your own skill

A skill is a small TypeScript plugin. Scaffold one (this runs in the sandbox and just writes files):

```bash
openshell sandbox exec -n shifty -- openclaw plugins init my-skill --name "My Skill" --directory /sandbox/my-skill
```

That creates a project — `openclaw.plugin.json` (the manifest), `src/index.ts` (your tool/skill code), `package.json`, and tests. You edit `src/index.ts` to register a tool, then **build and package** it:

```bash-ref
# Build produces ./dist/index.js, then packages it as a plugin:
npm install
npm run plugin:build        # = tsc, then `openclaw plugins build --entry ./dist/index.js`
openclaw plugins install ./my-skill
```

<Callout type="note" title="Build on the host or CI, install the artifact">
The sealed sandbox has no general outbound npm path (only the hosts in your policy), so a from-source <code>npm install</code> won't resolve a full toolchain inside it. In practice you <strong>build the plugin on your workstation or CI</strong> — where you have npm — and install the built result (a folder with <code>dist/</code>, a <code>.tgz</code>, or by publishing to ClawHub). That keeps the sandbox locked down while still letting you ship custom skills. The governance in steps 3–4 applies identically to your own skills.
</Callout>

<Callout type="rh" title="Skills are capabilities you can prove">
On most agent stacks, "install a plugin" means handing arbitrary code your whole network. Here, a skill is gated three ways — registry, enablement, and egress — and every byte it moves is in the audit log. That's the difference between an agent that's <em>extensible</em> and one that's extensible <em>and</em> governable.
</Callout>
