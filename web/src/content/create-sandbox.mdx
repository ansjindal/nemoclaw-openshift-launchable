You've met Shifty and configured it. Now create a **brand-new sandbox through the OpenShell gateway** — and watch the whole lifecycle. This is the real machinery: one command, and the gateway orchestrates a fully-isolated agent sandbox onto the cluster.

## The flow

<SandboxFlow />

This is **the power of OpenShell**: you don't hand-write Kubernetes YAML. You ask the **gateway** (gRPC :8080) for a sandbox, and it mints a scoped identity (a sandbox JWT), writes the `Sandbox` custom resource, the **agent-sandbox controller** reconciles it into a pod, and an **init-container side-loads the OpenShell supervisor** that wires up the sandbox's isolated network + policy. You get a ready, governed sandbox — not just a bare pod.

## 1. Create a sandbox via the gateway

The `openshell` CLI is already pointed at the in-cluster gateway. Create one:

```bash
openshell sandbox create --name scout -- sh -c 'echo ready; sleep 3600'
```

The gateway streams the lifecycle back: `Sandbox allocated` → it pulls the OpenShell **supervisor** image → the sandbox runtime starts. (The multi-GB sandbox image is pre-pulled by the launchable, so this is quick.)

## 2. Watch it become Ready

```bash
openshell sandbox list
```

You'll see `scout` move to **`Ready`**. Under the hood that's a real Kubernetes object — inspect what the gateway wrote for you:

```bash
oc -n openshell get sandbox scout -o jsonpath='{.metadata.labels}{"\n"}'
```

Notice the `openshell.ai/managed-by: openshell` label — that's how the gateway recognizes *its* sandboxes. And the pod the controller built:

```bash
oc -n openshell get pod scout -o jsonpath='{range .spec.initContainers[*]}init: {.name}{"\n"}{end}{range .spec.containers[*]}main: {.name}{"\n"}{end}'
```

The `openshell-supervisor-install` init-container is the side-load step from the diagram; the `agent` container is your workload, wrapped by the supervisor.

## 3. Run something inside it

It's a real, isolated sandbox. Exec into it the **native** way — through the gateway, which relays the command to the sandbox's supervisor:

```bash
openshell sandbox exec -n scout -- sh -c 'echo "hello from a gateway-built sandbox"; hostname; id'
```

That command never touched `oc` — the gateway brokered it end to end (mint identity → relay → run → stream back), which is the whole point of a control plane.

<Callout type="note" title="oc exec also works">
Since a sandbox is just a pod, <code>oc -n openshell exec scout -c agent -- …</code> reaches the same container via the Kubernetes API — handy when you want to bypass the gateway. Both land you in the same place.
</Callout>

## 4. Give your sandbox an identity

A sandbox is yours to shape. Inject configuration at creation with `--env`, or write files into its workspace. For an **OpenClaw** agent, an `IDENTITY.md` in the workspace *is* its persona — the same files you met in *Meet Shifty*:

```bash
oc -n openshell exec scout -c agent -- sh -c 'mkdir -p /workspace && cat > /workspace/IDENTITY.md <<MD
Name: Scout
Creature: a curious recon bot 🔭
Vibe: terse, precise, read-only
MD
cat /workspace/IDENTITY.md'
```

To spin a **fully custom agent**, point `--from` at the OpenClaw image and seed its `IDENTITY.md`/`SOUL.md` + config — the exact pattern the launchable uses for Shifty (`manifests/openclaw/workspace-seed/` + an initContainer):

```bash
# a customized agent = OpenClaw sandbox image + your own seeded identity/policy
openshell sandbox create --name recon \
  --from ghcr.io/nvidia/openshell-community/sandboxes/openclaw:latest \
  --env AGENT_PERSONA=recon -- sh -c 'sleep 3600'
```

<Callout type="tip" title="Your hackathon agents">
Each teammate can <code>openshell sandbox create</code> their <em>own</em> named agent with its own identity and policy — all governed by the same gateway. Give one a recon personality, another a remediation one.
</Callout>

## 5. Tear it down

```bash
openshell sandbox delete scout
openshell sandbox list
```

The gateway deletes the `Sandbox` CR, the controller removes the pod, and OpenShift reclaims everything.

<Callout type="rh" title="This is the whole model">
Ask the gateway for a sandbox → it mints an identity + writes a <code>Sandbox</code> CR → a controller reconciles it into a pod → the supervisor isolates it → OpenShift enforces RBAC/SCC/NetworkPolicy → you use it → delete to clean up. Every agent, including Shifty, rides this same path.
</Callout>

<Callout type="tip" title="Hackathon angle">
Your Cluster Copilot could <em>create sandboxes on demand</em> through the gateway — spin a scratch sandbox to run a risky command in isolation, then delete it. That's a few lines against the OpenShell API.
</Callout>
