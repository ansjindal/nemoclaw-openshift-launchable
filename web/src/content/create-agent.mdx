The platform is already running — a MicroShift cluster with the **OpenShell gateway** on it (Part II). Now you do the interesting part by hand: **create your own agent.**

In OpenShell, an agent is a **sandbox** — a pod the gateway creates, seals in its own private network, and governs with a policy. You ask for one with `openshell sandbox create`. We'll make one that runs **OpenClaw** (the agent framework). We'll go one small step at a time.

## Step 1 — Confirm the gateway is listening

Every command you run talks to the gateway. First check it answers:

```bash
openshell status
```

You want `Status: Connected` and a version. The endpoint reads `http://127.0.0.1:30808` — that's the gateway's NodePort, republished on this VM by a host forwarder (Part II explains why). If it's not connected, see [Troubleshooting](/learn/troubleshooting).

## Step 2 — Look at the policy you'll use

OpenShell is **deny-by-default**: a brand-new sandbox can reach **nothing** on the network. You grant access with a *policy* file. Read the template the repo ships:

```bash
cd ~/nemoclaw-openshift-launchable
cat policies/openclaw-sandbox.yaml
```

Two parts matter. First, the **filesystem/process** block — what the agent may touch on disk and run. Second, `network_policies`: a **map of named rules**, each saying *which binary* may reach *which host*. Here it's a single rule, `openclaw`, allowing the OpenClaw registry/API hosts over `:443` for `node`/`npm`. That's the whole grant — everything not listed is denied.

```yaml
network_policies:
  openclaw:
    endpoints:
      - { host: api.openclaw.ai, port: 443 }
      - { host: registry.npmjs.org, port: 443 }   # …and a couple more
    binaries:
      - { path: /usr/bin/node }
      - { path: /usr/local/bin/node }
```

<Callout type="note" title="Where's the inference host?">
Notice there's <strong>no rule for your inference endpoint</strong>. You don't need one: in the previous lesson you set the gateway's inference route, and the agent reaches the model at <code>https://inference.local</code> — a virtual host the supervisor intercepts <em>before</em> this policy is evaluated. The gateway makes the real upstream call (with your key). So the model just works, the real host never appears in the policy, and the agent never holds the key.
</Callout>

This policy needs no editing — use it as-is.

## Step 3 — Create the sandbox

One command creates the agent. Let's read each flag, then run it:

- `--name shifty` — what we'll call it.
- `--policy policies/openclaw-sandbox.yaml` — the egress grant from Step 2.
- `--from …/sandboxes/openclaw:latest` — the image. This OpenClaw sandbox image ships the `openclaw` agent **and** `iproute2` (the supervisor needs it to build the network seal).
- `-- sh -c 'sleep infinity'` — the **initial command**. `openshell sandbox create` runs whatever you put after `--`; by default it would drop into a shell and exit. We pass `sleep infinity` so the pod simply **stays alive and idle** — we'll start OpenClaw inside it ourselves in the next lesson. (It's a deliberate "keep the box running" no-op, not OpenClaw itself.)

```bash
openshell sandbox create --name shifty \
  --policy policies/openclaw-sandbox.yaml \
  --from ghcr.io/nvidia/openshell-community/sandboxes/openclaw:latest \
  -- sh -c 'sleep infinity'
```

The gateway allocates a pod, sideloads its **supervisor** (the per-pod component that seals the network and mints the sandbox's identity), and applies your policy.

## Step 4 — Wait until it's ready

List your sandboxes:

```bash
openshell sandbox list
```

The pod's container goes ready in seconds, but the gateway needs a moment longer while the supervisor finishes its handshake. The reliable "it's ready" signal is that you can run a command inside it:

```bash
openshell sandbox exec -n shifty -- echo "supervisor ready"
```

When that prints `supervisor ready`, your agent's home is live.

<Callout type="note" title="It's a pod, in the same namespace as the gateway">
<code>oc -n openshell get pod shifty</code> shows it as a normal pod — in the <strong>openshell</strong> namespace, right alongside the gateway (sandboxes are <em>not</em> in a separate namespace). But the agent runs in a <strong>private network namespace</strong> reachable only through the supervisor — that's the seal you'll explore in <a href="/learn/into-the-sandbox">Into a Sandbox</a>.
</Callout>

<Callout type="rh" title="What you just did">
One command, and the gateway turned it into a governed pod on OpenShift — deny-by-default networking, a scoped identity, your policy. Next: give it a brain and start OpenClaw.
</Callout>
