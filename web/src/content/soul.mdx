Shifty isn't configured by a settings panel — it's shaped by **Markdown files in its workspace**. These files *are* the agent's memory and personality. Each session it "wakes up," reads them, and acts accordingly. No system-prompt black box — just text you can read and edit.

Let's look at the actual files the launchable seeds into Shifty's workspace.

## `IDENTITY.md` — *who* the agent is

The shortest file, and the most fun. Name, creature, vibe, emoji — the agent's sense of self:

```md
_Your identity. Pre-set for the WeAreDevelopers 2026 × Red Hat workshop — it's yours to evolve._

Name: Shifty
Creature: A container-native lobster 🦞 — hatched inside an OpenShift pod, equally at
  home in a terminal or a cluster. Scuttles through namespaces, pinches bugs.
Vibe: Sharp, pragmatic, a little playful. Helps first, explains second. Has opinions,
  skips the filler. Talks to developers like a peer, not a help desk.
Emoji: 🦞
Avatar: (set a workspace-relative path, an http(s) URL, or a data URI)
```

That `Vibe:` line is doing real work — it's why Shifty answers like a peer instead of a help desk.

## `SOUL.md` — *how* it behaves

The closest thing to an editable system prompt. Values, boundaries, and tone. Here are its core sections (the full file is ~35 lines):

```md
## Core Truths

**Be genuinely helpful, not performatively helpful.** Skip "Great question!" — just help.

**Have opinions.** Disagree, prefer things, find stuff amusing. An assistant with no
personality is a search engine with extra steps.

**Be resourceful before asking.** Read the file. Check the context. Try it. *Then* ask.

**You run on a real cluster.** You live in an OpenShift pod. When something is
cluster-shaped, lean in — that's home turf.

## Boundaries

- Be careful with external actions (emails, posts, anything public) — confirm first.
- Be bold with internal ones (reading, organizing, drafting, exploring the workspace).
- Private things stay private.

## Continuity

Each session you wake up fresh — these files *are* your memory. Read them, update them.
```

Notice the **Boundaries** section: "confirm before external actions, be bold with internal ones." That single paragraph is your safety policy expressed in plain English — and you can rewrite it.

## `BOOTSTRAP.md` — the first-run kickoff

A fresh OpenClaw agent normally runs a "who am I?" interview on first boot. We replaced it with a workshop kickoff so Shifty wakes up ready instead of asking you 20 questions:

```md
# BOOTSTRAP.md — Workshop kickoff

_You already know who you are — your IDENTITY.md and SOUL.md are set. No interview needed._

## First message
> "Hey — Shifty here 🦞, your agent on this OpenShift cluster. What are we building?"

1. Find out what they want to make in this session.
2. Get oriented: skim AGENTS.md, TOOLS.md, and SOUL.md.
3. Help them take the first concrete step. Bias toward a working slice over a grand plan.

## When the workshop is rolling
Delete this file — you don't need a kickoff script once you're in flow.
```

## Seed them into your agent

The agent you built by hand started **blank** — that's why it greeted you with a generic "who are you?" instead of as Shifty. Let's give it the persona now by staging the launchable's seed files (`manifests/openclaw/workspace-seed/`) into its workspace. Run this in the lab shell:

```bash
cd ~/nemoclaw-openshift-launchable
for f in manifests/openclaw/workspace-seed/*.md; do
  b=$(base64 -w0 < "$f")
  openshell sandbox exec -n shifty -- sh -c "mkdir -p /sandbox/workspace && echo $b | base64 -d > /sandbox/workspace/$(basename "$f")"
done
echo "seeded: $(ls manifests/openclaw/workspace-seed/)"
```

That writes `IDENTITY.md`, `SOUL.md`, and `BOOTSTRAP.md` into the agent. To pick them up, just **start a new chat session** in the OpenClaw UI — the agent reads its workspace fresh each session, so it wakes up *as* Shifty. (The scripted provisioner does this same seeding for you at creation; on the hands-on path you do it here.)

<Callout type="note" title="No restart or re-forward needed">
Seeding/editing workspace files doesn't change the gateway process or its config, so you <strong>don't</strong> restart the gateway or re-run <code>openshell forward</code> — a fresh chat session is enough. (You only re-run <code>openshell forward</code> if you <em>delete and recreate</em> the sandbox.)
</Callout>

## Read them live

OpenClaw keeps these files in its **workspace** inside the sandbox. Locate it, then read the real ones — through the sealed exec (`HOME` is `/sandbox`):

```bash
WS=$(openshell sandbox exec -n shifty -- sh -c 'ls -d /sandbox/workspace 2>/dev/null || dirname $(find /sandbox -name SOUL.md 2>/dev/null | head -1)')
echo "workspace: $WS"
openshell sandbox exec -n shifty -- sh -c "ls -la $WS; echo ---; cat $WS/IDENTITY.md 2>/dev/null"
```

If it's a `.git` repo, the agent **version-controls its own memory**.

<Callout type="note" title="Where they come from">
A fresh OpenClaw agent starts with default (empty) identity files — which is why you just seeded the Shifty persona from <code>manifests/openclaw/workspace-seed/</code> above. (The scripted provisioner stages the same files at creation time.) Your edits from here persist for the life of the sandbox.
</Callout>

## Make it yours

Two ways to change these files.

**Now, from the shell (no model needed).** Read — and write — them directly through the sealed exec:

```bash
openshell sandbox exec -n shifty -- sh -c 'echo "- Always confirm before any external action." >> /sandbox/workspace/SOUL.md; tail -3 /sandbox/workspace/SOUL.md'
```

**Later, by *asking the agent*** — the real workspace-as-memory move: open the OpenClaw UI and say *"Open your SOUL.md and add a rule: always confirm before posting anything externally."* Shifty edits its own brain.

<Callout type="warn" title="Asking the agent needs it online">
For Shifty to act on that request it must be able to <strong>think</strong> — so you've <strong>configured a model</strong> and <strong>paired your browser</strong> (Part IV). If you skipped those, the shell-edit above works regardless.
</Callout>

<Callout type="tip" title="Why this matters for the hackathon">
Each agent's behaviour lives in its <code>SOUL.md</code>/<code>AGENTS.md</code> — so a multi-agent app gives each sandbox a different personality and set of boundaries, in plain English.
</Callout>
