Out of the box your agent can *reason*, but it can't *look things up* — it has no search tool, and even if it did, deny-by-default networking would block it. Let's give it **web search**, for free, with no API key: OpenClaw 2026.6 ships a bundled **DuckDuckGo** plugin that scrapes DuckDuckGo's no-JS HTML results. Two things have to line up — the **egress policy** (OpenShell) and the **plugin** (OpenClaw).

## 1. The policy already allows it

Search means the agent's `node` process reaches DuckDuckGo's hosts. Our sandbox policy has a dedicated rule for exactly that — look at the `websearch` block:

```bash
cd ~/nemoclaw-openshift-launchable
sed -n '/websearch:/,/node }/p' policies/openclaw-sandbox.yaml
```

```yaml
websearch:
  endpoints:
    - { host: duckduckgo.com, port: 443, access: full }
    - { host: html.duckduckgo.com, port: 443, access: full }
    - { host: lite.duckduckgo.com, port: 443, access: full }
  binaries:
    - { path: /usr/bin/node }
    - { path: /usr/local/bin/node }
```

`access: full` matters here: an HTTPS scrape is a raw TLS tunnel, so the rule grants a full L4 connection (not a parsed REST allow-list) to **only** those three hosts, for **only** `node`. Everything else stays denied.

## 2. Turn the DuckDuckGo plugin on

Two config keys: enable the bundled plugin, and pick it as the web-search provider. Then **restart the gateway** so it loads the plugin:

```bash
openshell sandbox exec -n shifty -- sh -c 'openclaw config set plugins.entries.duckduckgo.enabled true; openclaw config set tools.web.search.provider duckduckgo; openclaw config get tools.web.search'
```

<Callout type="note" title="One line, on purpose">
Keep that as a <strong>single line</strong> (commands joined with <code>;</code>). <code>openshell sandbox exec -- sh -c '…'</code> rejects a quoted argument that contains literal newlines, so a multi-line block fails with <em>"command argument contains newline"</em>.
</Callout>

You'll see `Updated … Restart the gateway to apply.` and the provider read back as `duckduckgo`. Restart (same command as before):

```bash
openshell sandbox exec -n shifty -- sh -c \
  'pkill -f "openclaw gateway run"; sleep 1; cd /sandbox && setsid nohup openclaw gateway run --port 30789 --bind lan --auth password --password openshell-wad26 --allow-unconfigured >/sandbox/gateway.log 2>&1 < /dev/null & sleep 6; grep -E "listening on ws" /sandbox/gateway.log | tail -1'
```

## 3. Make it search

Ask the agent something it can only answer by looking it up. We run one embedded turn:

```bash
openshell sandbox exec -n shifty -- sh -c 'cd /sandbox && openclaw agent --local --agent main -m "Use web search to find the official OpenShift documentation site and reply with just the domain." --session-id wsdemo'
```

It comes back with **`docs.openshift.com`** — fetched live from DuckDuckGo, through the policy, by your sealed agent. Try it in the **Control UI** too: ask *"what's new in OpenShift 4.20?"* and watch it search.

<Callout type="note" title="Free, but still governed">
DuckDuckGo needs no key, which is why it's the friendly default. But notice the agent didn't get "the internet" — it got <strong>three hosts</strong>. If you swapped in a provider that hit <code>api.bing.com</code>, the call would be <em>denied</em> until you added that host to the policy. The search tool is only ever as reachable as your egress rules allow.
</Callout>

<Callout type="rh" title="A capability you can audit">
Every search is a `node → html.duckduckgo.com` connection in <code>openshell logs shifty</code> — allowed, attributed, and bounded. That's the whole point: you can hand an agent real-world reach without handing it the open network.
</Callout>
